## 🎯 元认知 Q（Meta-Q）

### 我对"融合"的理解正确吗？

**当前状态分析**：
- 6 个 3Q 相关技能分散存在
- 依赖关系复杂（self-challenge-3q → quality-prevention-milestone → 3Q-Plus-v3 → quality-os → quality-os-trigger）
- 用户需要记住多个触发词和场景

**融合的真正需求**：
1. **用户视角**：一个入口，自动匹配最合适的 3Q 检查
2. **维护视角**：减少重复代码，统一更新
3. **性能视角**：减少技能调用链长度

**我的假设验证**：
- ✅ 假设 1：用户希望简化使用流程
- ✅ 假设 2：当前技能数量过多（6 个）
- ⚠️ 假设 3：融合后性能会提升（需要验证）

**理解确定性**: 85%（还需要用户确认）

## ⚔️ 挑战 Q（Challenge-Q）

### 如果融合方案完全错误，会怎样？

**最强反对意见**：

#### 反对 1：融合会损失灵活性
```
当前：6 个技能可以独立使用、独立更新
融合后：单一技能，修改影响所有场景

反驳**：
- 可以通过配置化保留灵活性
- 模块化设计，各检查框架独立

#### 反对 2：增加复杂度
```
当前：触发词明确，场景清晰
融合后：需要智能识别，增加判断逻辑

反驳**：
- 用户侧简化（一个入口）
- 内部复杂度可控（规则引擎）

#### 反对 3：性能下降
```
当前：直接调用对应技能
融合后：先识别再调用，增加一层

反驳**：
- 识别逻辑轻量（<100ms）
- 减少用户选择时间（>1 分钟）
- 净收益：正

## 🏅 竞争 Q（Competition-Q）

### 与现有方案比有什么独特优势？

**对比分析**：

| 方案 | 技能数量 | 触发词 | 灵活性 | 维护成本 | 用户体验 |
|------|----------|--------|--------|----------|----------|
| **当前方案** | 6 个 | 20+ 个 | 高 | 高 | 中 |
| **完全融合** | 1 个 | 1-2 个 | 低 | 低 | 高 |
| **分层融合**（推荐） | 2 个 | 3-4 个 | 中高 | 中 | 高 |

**推荐方案：分层融合**

```
Layer 1: 3Q（统一入口）
├─ 自动识别文档类型 + 重要性
├─ 调用对应检查框架
└─ 返回标准化结果

Layer 2: 3Q-Framework（检查框架库）
├─ 3Q-Lite（三问，日常文档）
├─ 3Q-Pro（六问，重要文档）
├─ 3Q-Max（三阶段，里程碑）
└─ 可独立调用
```

**差异化优势**：
1. ✅ 统一入口 + 保留灵活性
2. ✅ 智能识别 + 可手动指定
3. ✅ 向后兼容（旧触发词仍可用）

## 🎯 融合方案设计

### 方案名称：3Q-Unified v1.0

### 架构设计

```
┌─────────────────────────────────────────────────────────┐
│                      3Q（统一入口）                      │
│  触发词："3Q 检查", "quality check", "质量评分"           │
└────────────────────┬────────────────────────────────────┘
                     │
         ┌───────────┴───────────┐
         │  智能识别引擎          │
         │  - 文档类型检测        │
         │  - 重要性评估          │
         │  - 场景匹配            │
         └───────────┬───────────┘
                     │
    ┌────────────────┼────────────────┐
    │                │                │
┌───┴────┐    ┌─────┴─────┐   ┌────┴────┐
│ 3Q-Lite│    │  3Q-Pro   │   │ 3Q-Max  │
│ (三问) │    │  (六问)   │   │ (三阶段)│
│        │    │           │   │         │
│ 日常   │    │  重要     │   │ 里程碑  │
│ <10 分  │    │  10-30 分 │   │ >30 分  │
└────────┘    └───────────┘   └─────────┘
```

### 识别规则

| 文档类型 | 重要性 | 推荐检查 | 预计时间 |
|----------|--------|----------|----------|
| 日常笔记 | 低 | 3Q-Lite | 5-10 分钟 |
| 会议纪要 | 低 | 3Q-Lite | 5-10 分钟 |
| PRD/方案 | 高 | 3Q-Pro | 30-45 分钟 |
| 技能文档 | 高 | 3Q-Pro | 30-45 分钟 |
| 里程碑节点 | 高 | 3Q-Max | 2-4 小时 |
| 产品发布 | 高 | 3Q-Max | 2-4 小时 |

### 触发词设计

**主触发词**（推荐）：
- "3Q 检查"
- "quality check"
- "质量评分"

**保留触发词**（向后兼容）：
- "self-challenge"
- "3Q Plus"
- "milestone 检查"
- "QualityOS"

### 配置化设计

```yaml
# 3Q 配置示例
3Q:
  auto_detect: true  # 自动识别
  manual_override: true  # 允许手动指定
  
  rules:
